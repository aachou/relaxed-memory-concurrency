# 《共享内存一致性模型：教程》详细总结

**作者：** Sarita V. Adve, Kourosh Gharachorloo  
**原文：** `docs/shared-memory-consistency-models-tutorial.md`（中英双语）

---

## 一、论文背景与动机

并行共享内存在 90 年代中期被广泛接受，但**内存一致性模型（Memory Consistency Model）** 这个接口层的规范问题一直缺乏清晰的系统化梳理。最直观的模型——**顺序一致性（Sequential Consistency, SC）**——要求所有内存操作按某个全局顺序逐个执行，且单处理器的操作必须按程序顺序出现。这虽然编程简单，却严重限制了单处理器中常用的硬件和编译器优化（写缓冲、流水线、编译代码移动等）。工业界开始采用更宽松的模型（如 Digital Alpha、SPARC V8/V9、IBM PowerPC），但这些模型术语混乱、定义各异，导致程序员和系统设计者难以理解和比较。

**论文目标：** 用统一简单的术语，以系统为中心（system-centric）的视角来描述 SC 及多种宽松模型，兼顾以程序员为中心（programmer-centric）的视角。

## 二、顺序一致性（SC）的详细要求

Lamport 的形式化定义：**任何执行的结果与所有处理器的操作按某种顺序执行、且每个处理器的操作按其程序指定顺序出现在该序列中的结果相同。**

SC 的**两个核心要求**：

| 要求 | 含义 | 硬件影响 |
|------|------|----------|
| **程序顺序要求** | 处理器必须确保前一个内存操作完成，才能发出按程序顺序的下一个操作 | 等待写确认、不可绕过写缓冲 |
| **写原子性要求** | 写操作对所有处理器同时可见；同一位置的写操作必须被所有人看到相同的顺序 | 需要缓存一致性协议 + 确认响应 |

## 三、SC 与优化的冲突（Section 5）

论文用三个典型优化来说明为何 SC 过于严格：

1. **写缓冲绕过（Write Buffer Bypassing）**——突破 write→read 程序顺序，导致 Dekker 算法中两个处理器都读到 0。
2. **写操作重叠（Overlapping Writes）**——突破 write→write 程序顺序，导致 Data 和 Head 的写入顺序错乱。
3. **非阻塞读（Non-Blocking Reads）**——突破 read→read/write 程序顺序，导致 P2 读到 Data 旧值。

这些优化在单处理器中完全安全（不违反数据/控制依赖），但在多处理器中会破坏 SC。

**缓存系统引入的额外问题：**
- 需要缓存一致性协议传播写值
- 写完成检测需要确认响应（不仅是到达内存模块，还需等失效/更新被目标缓存确认）
- 写原子性更难维持：需要对同一位置的写串行化，且读操作不能在新值传播到所有缓存前返回该值

## 四、宽松模型分类体系（Section 6）

论文提出一个**二维分类框架**：

| 放宽维度 | 允许的优化 | 对应 SC 要求 |
|----------|------------|-------------|
| write→read 放宽 | 写缓冲绕过 | 程序顺序 |
| write→write 放宽 | 写操作流水线/重叠 | 程序顺序 |
| read→read/write 放宽 | 非阻塞读 | 程序顺序 |
| 写原子性放宽 | 提前返回其他处理器的写值 | 写原子性 |
| 提前读取自身写值 | 写缓冲转发到后续读 | 两者兼有 |

### 模型图谱

| 模型 | 放宽类型 | 安全网 |
|------|----------|--------|
| **SC** | 无 | 无（本身就是最严格） |
| **IBM 370** | write→read | 串行化指令（serialization instructions） |
| **TSO** (SPARC V8) | write→read + 提前读自身写 | 读-修改-写（RMW）操作 |
| **PC** (Processor Consistency) | write→read + 提前读任何写 | RMW 操作 |
| **PSO** (SPARC V8) | write→read + write→write | STBAR + RMW |
| **WO** (Weak Ordering) | 全部程序顺序 | 同步操作分类（sync/data） |
| **RCsc** (Release Consistency) | 全部程序顺序 | acquire/release + special 操作 |
| **RCpc** | 全部程序顺序（特殊操作间保留 write→read） | acquire/release + RMW |
| **Alpha** | 全部程序顺序 | MB / WMB 指令 |
| **RMO** (SPARC V9) | 全部程序顺序 | MEMBAR（4 位编码） |
| **PowerPC** | 全部程序顺序 + 写原子性 | SYNC + RMW |

**关键洞察：** 只有放宽**所有**程序顺序的模型才能给编译器足够大的优化空间——因为读写在程序中细粒度交错，仅放宽 write→read 对编译器几乎无帮助。

**特别值得注意的模型差异：**

- **RCsc vs RCpc**：RCsc 在特殊操作之间维持 SC，RCpc 则放宽了特殊 write→special read 的顺序——这是一个微妙但影响原子性的差异。
- **Alpha vs PowerPC**：Alpha 提供单独的 MB 和 WMB 指令，且不需要写原子性安全网；PowerPC 只有一个 SYNC 指令，且在相同地址的两读之间放置 SYNC 仍不能保证顺序，需要 RMW。

## 五、以程序员为中心的视角（Section 7）

论文指出系统中心视角的缺陷：程序员必须理解硬件重排和原子性优化才能推理正确性。

**DRF0（无数据竞争）方法：**
- 程序员只需在**顺序一致性执行**下识别哪些操作之间存在竞争（race）
- 涉及竞争的操作标记为**同步（synchronization）**，保守处理
- 不涉及竞争的操作标记为**数据（data）**，可积极优化
- 不确定时可选"不知道"（don't-know），保守标记为同步

**信息传递机制：**
- 语言层面：doall 循环隐式传递；同步库调用隐式传递；显式标注（类型声明/区域标记）
- 硬件层面：特定页/地址范围；额外操作码；虚拟地址高比特位阴影（address shadowing）；编译器将标注转为栅障指令

## 六、核心结论与历史意义

1. **宽松模型确实比 SC 性能更好**[9, 11, 6]，且处理器速度相对内存/通信速度的增长只会加大收益
2. 当今几乎所有架构都支持某种形式的栅障指令
3. 宽松模型的**最大代价是编程复杂性**——但这可以通过以程序员中心的高级抽象来缓解
4. 语言标准化（如 High Performance Fortran 的 copy-in/copy-out 语义）提供了不同于 SC 的内存模型
5. "最佳内存一致性模型"的选择远未解决，需要语言设计者和硬件设计者更密切的合作

**历史影响：** 这篇论文用统一框架系统化了此前混乱的宽松模型领域，其分类方法和术语（TSO/PSO/RCsc/RCpc/WO 等）至今仍是计算机体系结构教材的标准内容。论文提出的"以系统为中心 vs 以程序员为中心"的双重视角为后续 C++11/C11 内存模型、Java 内存模型、以及 RISC-V 内存模型的设计提供了理论基础。
