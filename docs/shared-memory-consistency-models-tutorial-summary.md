# 共享内存一致性模型

**作者**：Sarita V. Adve & Kourosh Gharachorloo

---

## 1. 共享内存并发带来的问题

多处理器中，**一个读操作应该返回什么值？**

- 单处理器："最后一次写入"由**程序顺序**定义
- 多处理器：不同处理器上的操作之间没有程序顺序，语义模糊

**内存一致性模型** = 程序员与系统之间的契约，规定内存如何表现。

---

## 2. 顺序一致性 (SC)

**Lamport 定义 (1979)**：所有内存操作按某种全局顺序逐个执行，每个处理器的操作按程序顺序出现。

**两个要求**：
1. **程序顺序** — 单处理器内操作按序
2. **写原子性** — 写操作看起来对所有处理器瞬时完成

**经典反例 — Dekker 算法**：

```
P1: Flag1=1; if(Flag2==0) CS
P2: Flag2=1; if(Flag1==0) CS
```
SC 保证不会双双读到 0。任何优化若允许写→读重排，就会打破这个保证。

---

## 3. 为什么 SC 难以实现？

### 3.1 硬件优化破坏 SC

| 优化 | 破坏 | 例子 |
|------|------|------|
| 写缓冲绕过 | 写→读顺序 | Dekker 中双读 0 |
| 写操作重叠 | 写→写顺序 | P2 看到 Head 新值但 Data 旧值 |
| 非阻塞读 | 读→读/写顺序 | 读操作交错导致不一致 |

### 3.2 缓存引入更多问题

缓存一致性 ≠ 内存一致性：
- 缓存一致性只保证同一位置的写对所有处理器顺序一致
- SC 要求所有位置的写顺序一致 + 程序顺序

**写完成检测**：写操作需等待所有缓存副本确认失效/更新。

**写原子性**：传播到多副本本质上是非原子的，需要：
1. 同位置写串行化
2. 读不能提前返回写值（直到所有副本确认）

### 3.3 编译器也破坏 SC

- 代码移动、寄存器分配、公共子表达式消除等都会重排/消除共享内存操作
- 例：把 Head 分配到 P2 的寄存器中 → 循环永不终止

## 4. 宽松内存模型

### 分类框架

按两个维度分类：
1. **程序顺序放宽**：写→读 / 写→写 / 读→读/写（仅不同地址）
2. **写原子性放宽**：是否允许提前读到他人写

### 模型全景

| 模型 | 写→读 | 写→写 | 读→读/写 | 原子性放宽 | 安全网 |
|------|:---:|:---:|:--------:|:---------:|--------|
| **SC** | ✗ | ✗ | ✗ | ✗ | — |
| **IBM 370** | ✓ | ✗ | ✗ | ✗ | 串行化指令 |
| **TSO** | ✓ | ✗ | ✗ | ✗ | RMW |
| **PC** | ✓ | ✗ | ✗ | ✓ | RMW |
| **PSO** | ✓ | ✓ | ✗ | ✗ | STBAR + RMW |
| **WO** | ✓ | ✓ | ✓ | ✗ | 同步标记 |
| **RCsc** | ✓ | ✓ | ✓ | 可能 | acquire/release |
| **RCpc** | ✓ | ✓ | ✓ | ✓ | acquire/release + RMW |
| **Alpha** | ✓ | ✓ | ✓ | ✗ | MB / WMB |
| **RMO** | ✓ | ✓ | ✓ | ✗ | MEMBAR |
| **PowerPC** | ✓ | ✓ | ✓ | ✓ | SYNC + RMW |

### 核心模型详解

#### TSO (SPARC V8)
写→读放宽，但禁止提前读他人写。x86 的 TSO 模型与此类似。

#### PSO (SPARC V8)
写→读 + 写→写放宽。不同地址的写可流水线化。

#### WO (弱排序)
操作分类：**数据** vs **同步**。同步操作之间可自由重排数据操作。

#### RCsc/RCpc (释放一致性)
更细的分类链：普通 → 特殊 → sync → acquire/release。
- **acquire**（如 lock）：之前所有操作完成后才能执行
- **release**（如 unlock）：之前所有操作必须在 release 之前完成
- RCsc：特殊操作间保持 SC
- RCpc：放宽特殊写→特殊读

#### Alpha / RMO / PowerPC
通过显式 fence 指令提供安全网（而非操作分类）。

## 5. 程序员中心视角

### 问题
以系统为中心的模型把底层优化暴露给程序员，编程复杂。

### 方法（DRF0）
程序员标记哪些操作是**同步**操作（涉及竞争的操作），系统保证标记正确时表现为 SC。

**竞争定义**：在 SC 执行中，两个操作访问同一位置、至少一个是写、之间无其他操作插入。

**逐步调优**：
- 全部标记为同步 → 正确但性能低
- 精确标记性能关键区域 → 渐进优化

### 信息传递
- 语言级：doall 循环（隐式）、同步库（隐式）、类型声明（显式）
- 硬件级：地址范围映射、多种指令 opcode、地址阴影、或最终转为 fence

---

## 6. 结论

- 宽松模型提供比 SC 更好的性能（硬件 + 编译器优化）
- 代价是编程复杂度增加
- 商业架构已广泛支持（x86 TSO、ARM、PowerPC、RISC-V）
- 趋势：更高级的语言抽象（如 C++ 内存模型）使程序员只需关心 happens-before 关系，而非底层重排细节

---

**一句话总结**：内存一致性模型定义多处理器中读操作能看到什么值。SC 直观但限制优化，宽松模型释放性能但增加复杂度。理解这些模型的本质差异是编写正确并行程序的基础。
