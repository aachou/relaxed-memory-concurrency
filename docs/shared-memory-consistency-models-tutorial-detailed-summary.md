# 共享内存一致性模型——详细总结

> **原标题：** Shared Memory Consistency Models: A Tutorial  
> **作者：** Sarita V. Adve, Kourosh Gharachorloo  
> **关键词：** 内存一致性模型、顺序一致性、宽松一致性、缓存一致性、释放一致性、DRF0  
> **原文摘要：** 并行系统中共享内存抽象的正确高效使用需要内存语义的形式化规范。本文以统一术语描述从顺序一致性到多种宽松模型的谱系，兼顾以系统为中心和以程序员为中心的两种视角。

---

## 1. 引言

### 1.1 共享内存 vs 消息传递

共享内存（单地址空间）抽象相比消息传递具有两个关键优势：
- **从单处理器自然过渡**：程序员继续使用 load/store 思维，无需学习发送/接收原语
- **简化数据划分和动态负载分布**：数据天然对所有处理器可见，无需显式迁移

### 1.2 问题：读操作应返回什么值？

以 LocusRoute (SPLASH) 为例：P1 分配任务记录 → 写入 Data 字段 → 将任务插入队列 → 更新 Head 指针。其他处理器等待 Head → 出队 → 读 Data。若内存系统允许 P1 的写 Data 和写 Head 以不同顺序对其他处理器可见，则出队处理器可能读到旧 Data 值。

**内存一致性模型**正是解决这一问题的形式化规范——它定义了读操作在给定执行中可返回的值集。

## 2. 谁应该关心内存一致性模型？

三类受影响的人群：

| 角色 | 关注点 |
|------|--------|
| **程序员** | 可编程性：能否用直觉推理程序正确性？ |
| **系统设计者（硬件+编译器）** | 性能：允许哪些优化？实现复杂度如何？ |

每个接口层次（机器码、高级语言）都有对应的内存模型规范。

## 3. 单处理器的内存语义

单处理器顺序语义：读操作返回"最后一次"写入的值，"最后一次"由**程序顺序**精确定义。

单处理器之所以能高效支持顺序语义，是因为只需维护**数据依赖**和**控制依赖**：
- 同一位置的操作不能重排
- 一个操作控制另一操作执行时不能重排
- **不同位置的操作可以自由重排**

这允许的优化包括：
- **硬件**：流水线、多发射、写缓冲绕过和转发、无锁缓存
- **编译器**：寄存器分配、代码移动、循环变换

**多处理器的根本困难**：不同处理器的操作之间不存在程序顺序，但 SC 要求全局顺序和原子性——这与上述优化直接冲突。

## 4. 理解顺序一致性（SC）

### 4.1 正式定义（Lamport）

> 一个多处理器系统是顺序一致的，如果任何执行的结果都与所有处理器的操作按**某个顺序**执行、且每个处理器的操作按其程序出现在该序列中的结果相同。

### 4.2 SC 的两个方面

1. **程序顺序**：单个处理器的操作按程序顺序出现在全局序列中
2. **原子性（写原子性）**：所有操作看起来是瞬时完成的——写操作的效果同时对所有处理器可见

### 4.3 两个经典示例

**示例 1：Dekker 算法（程序顺序的重要性）**

```
P1: Flag1 = 1; if (Flag2 == 0) enter CS;
P2: Flag2 = 1; if (Flag1 == 0) enter CS;
```

SC 保证两个读操作不会都返回 0。如果写→读顺序被破坏（如写缓冲绕过），双处理器都将进入临界区。

**示例 2：三级写传播（原子性的重要性）**

```
P1: A = 1
P2: r1 = A; B = 1;
P3: r2 = B; r3 = A;
```

在 SC 下，若 P2 读到 A=1 并写入 B=1，则 P3 读到 B=1 后必须也读到 A=1。若写操作不是原子的，P3 可能看到 A 的旧值。

## 5. 实现顺序一致性的挑战

### 5.1 无缓存架构的硬件优化 vs SC

| 优化 | 违反 SC 的方式 | 图例 |
|------|---------------|------|
| **写缓冲绕过** | 读越过缓冲中未完成的写 → 两个读都看到 0（Dekker 失败） | 图 5(a) |
| **写操作重叠** | 对不同地址的两个写可能乱序完成 → P2 看到 Head 新值但 Data 旧值 | 图 5(b) |
| **非阻塞读** | 读操作与后续读/写重叠 → 读 Data 和 Head 的到达顺序错乱 | 图 5(c) |

### 5.2 缓存架构的额外复杂性

#### 5.2.1 缓存一致性 ≠ 内存一致性

缓存一致性协议通常保证：
1. 写操作最终对所有处理器可见
2. 同一位置的写操作对所有处理器顺序一致（串行化）

但这**不足以保证 SC**，因为 SC 要求**所有位置**的写操作全局顺序一致，且要求程序顺序。

论文将缓存一致性协议视为**传播机制**（通过失效或更新），内存一致性模型则是决定新值何时对给定处理器可见的**策略**。

#### 5.2.2 检测写操作完成

有缓存时，写操作完成的条件比无缓存更严格：
- 需要对所有缓存副本发送失效/更新消息
- 需要等待所有目标缓存回复确认
- 常见优化：**提前确认**（收到消息即确认，在真正应用之前）——只要保证入队消息按序处理，就是安全的

#### 5.2.3 维持写原子性假象

需要两个条件：

**条件 1：同一位置的写操作串行化**

如果 P1 写 A=1 和 P2 写 A=2 的更新消息以不同顺序到达 P3 和 P4，则两个处理器看到 A 的值不同，违反原子性。解决方案：
- **目录串行化**：所有更新/失效由单一目录节点转发
- **确认串行化**：当前写操作的所有更新被确认后，才允许下一写操作

**条件 2：写操作对所有处理器可见后才允许读**

如果 P2 读到了 A=1（P1 写入），但 P3 尚未收到 A 的更新，P3 可能读到旧值 A=0。解决方案：
- 失效协议：自然满足（读前先获得所有权）
- 更新协议：需要**两阶段更新方案**
  - 阶段 1：发送更新值，等待确认，**不允许读新值**
  - 阶段 2：发送"放行"消息，收到后才允许读新值

### 5.3 编译器限制

SC 对编译器的约束与对硬件的约束类似：
- 共享内存操作不能重排（禁止代码移动、软件流水线中的重排）
- 寄存器分配不能将共享变量提升到寄存器（否则其他处理器的写入永远不可见）

示例：图 5(b) 中，若 P2 将 Head 读入寄存器并在循环中复用寄存器值，则 P1 对 Head 的写入对 P2 永远不可见，循环永不终止。

### 5.4 SC 的缓解技术

1. **自动预取所有权**：对写缓冲中延迟的写操作发预取独占请求，部分重叠延迟
2. **推测读服务**：延迟的读操作先推测执行，若缓存行在真正该发出时被失效则回滚
3. **Shasha-Snir 编译器算法**：分析哪些操作对可以安全重排（指数复杂度，实际偏保守）

## 6. 宽松内存模型

### 6.1 分类方法论

论文从两个维度对模型分类：

**维度 1：程序顺序放宽**
- 写 → 读（不同地址）
- 写 → 写（不同地址）
- 读 → 读/写（不同地址）

**维度 2：写原子性放宽**
- 是否允许读操作在写操作对所有处理器可见之前返回其值

此外还单独列出**提前读自身写操作**的放宽——这在 SC 下也允许，因此不视为模型间差异。

### 6.2 放宽写→读程序顺序

#### IBM 370
- 放宽写→读顺序
- **不**放宽写原子性（读不能返回自身写值，直到写对所有处理器可见）
- 安全网：串行化指令（compare&swap 等）

#### SPARC V8 TSO (Total Store Order)
- 放宽写→读顺序
- 部分放宽写原子性：允许读返回自身写值，但**不**允许读返回其他处理器写值直到全局可见
- 安全网：RMW 操作（通过 dummy write 恢复顺序）

#### PC (Processor Consistency)
- 放宽写→读顺序
- 完全放宽写原子性：允许读返回任意处理器写值，即使写尚未全局可见
- 安全网：RMW 操作（但不如 TSO 强——仅写→读中的读端需是 RMW 才有效）

**三种模型对比**：

| 场景 | IBM 370 | TSO | PC |
|------|---------|-----|----|
| 读自身未完成的写 | 阻塞 | 允许 | 允许 |
| 读另一处理器未完成的写 | 阻塞 | 阻塞 | 允许 |
| Dekker 算法（图 5(a)）SC 结果 | ✓ 保证 | ✗ 可能失败 | ✗ 可能失败 |
| 三级传播（图 4(b)）SC 结果 | ✓ 保证 | ✓ 保证 | ✗ 可能失败 |

### 6.3 放宽写→读 + 写→写程序顺序

#### SPARC V8 PSO (Partial Store Order)
- 在 TSO 基础上进一步放宽写→写顺序
- 同地址的两个写仍保序（这是缓存一致性协议的保证），不同地址的写可乱序完成
- 安全网：STBAR 指令（在两个写之间插入） + TSO 的 RMW 机制
- 实现：FIFO 写缓冲 + 计数器（发出写+1，收到确认-1，归零表示所有先前写完成）

### 6.4 放宽全部程序顺序

这组模型允许对所有不同地址的操作任意重排，是性能收益最大的一类。

#### 6.4.1 WO (Weak Ordering)
- 操作分为**数据操作**和**同步操作**
- 两个同步操作之间的数据操作可任意重排
- 同步操作之间保持程序顺序
- 安全网：同步操作 = 天然 fence（发出前等待所有操作完成，完成后才发出后续操作）
- 写原子性：始终对程序员保持原子

#### 6.4.2 RC (Release Consistency) — RCsc 和 RCpc

操作分类层次（三级）：

```
所有操作
 ├── 普通 (ordinary) — 对应 WO 的数据操作
 └── 特殊 (special)
      ├── nsync — 异步数据操作，不用于同步
      └── sync
           ├── acquire — 为获取共享位置访问权的读（如 lock）
           └── release — 为授予共享位置访问权的写（如 unlock）
```

**RCsc** 约束：acquire → all, all → release, special → special（特殊操作间维持 SC）

**RCpc** 约束：同上，但 special → special 中排除特殊写→特殊读（类似 PC 的写→读放宽）

RCsc 与 RCpc 的关键区别：

| 特性 | RCsc | RCpc |
|------|------|------|
| 特殊操作间顺序 | SC（全局保序） | PC（写→读可重排） |
| 写原子性 | 复杂实现可破坏，程序员**不能**依赖 | 默认非原子，需 RMW 恢复 |
| 更高级抽象 | 附带 PL（properly-labeled）模型 | 同 RCsc |

#### 6.4.3 Alpha
- 两种栅障：**MB**（全序）、**WMB**（仅写→写）
- 写原子性始终保持
- 对同一地址的两个读也可能重排？MB 保证读→读顺序

#### 6.4.4 SPARC V9 RMO (Relaxed Memory Order)
- MEMBAR 指令：4 位编码，组合指定 读→读、读→写、写→读、写→写
- 相比 TSO/PSO 的进步：不再需要 RMW 来恢复写→读顺序（MEMBAR 即可）
- 写原子性始终保持

#### 6.4.5 PowerPC
- 单一 SYNC 指令（类似 Alpha MB）
- **重要特性**：即使在两个对同一地址的读之间放 SYNC，第二次读仍可能返回比第一次更旧的值！
  - 需要 RMW 来保证同一地址读→读顺序
- 写原子性**不保持**（允许早见），类似 PC/RCpc 需 RMW

### 6.4.6 编译器优化

只有**放宽全部程序顺序**的模型才对编译器有意义。原因：程序的读写通常是细粒度交错的，仅放宽写→读或写→写不足以让编译器获得实质性的重排空间。

在 WO/RC/Alpha/RMO/PowerPC 下，编译器在 fence 或同步操作之间可以获得与单处理器相同的优化自由度。

### 6.5 各种模型与商用系统对照

| 架构 | 支持模型 |
|------|---------|
| IBM 370 | IBM 370 |
| Sun SPARC V8 | TSO（默认）、PSO |
| Sun SPARC V9 | RMO（默认）、TSO（兼容） |
| DEC Alpha | Alpha 模型 |
| IBM PowerPC | PowerPC 模型 |
| Intel x86 | 接近 TSO |
| ARM | 类似 RCsc（带数据依赖保序） |

## 7. 以程序员为中心的视角

### 7.1 动机

以系统为中心的规范将底层重排优化暴露给程序员，导致编程复杂度高。以程序员为中心的规范要求程序员提供**程序级信息**，由系统保证优化不破坏正确性。

### 7.2 DRF0（无数据竞争）框架

**核心思想**：顺序一致性是可取的"正确性"定义（因为它是最自然的单处理器推广）。程序员需标记出所有可能参与**竞争**的操作。

**竞争的定义**：在一个顺序一致执行中，两个操作访问同一位置、至少有一个是写、且两个操作之间没有任何其他操作插入。

**标记规则**：
- 参与竞争的操作 → 标记为**同步操作**（保守执行）
- 不参与竞争的操作 → 可标记为**数据操作**（积极优化）
- 不确定 → 标记为同步（"don't know"选项），先确保正确性，再逐步调优

**正确标记的保证**：程序行为等价于顺序一致性。

### 7.3 信息传递机制

#### 7.3.1 语言级

| 机制 | 示例 | 信息传递方式 |
|------|------|------------|
| doall 循环 | Fortran doall | 隐式无竞争（迭代间无数据依赖） |
| 同步库 | pthread_mutex_lock | 隐式同步（调用内部的操作自动标记） |
| 显式声明 | C++ `memory_order` | 程序员显式标记 |
| 变量类型 | volatile 或 sync 类型 | 按地址范围标记 |

#### 7.3.2 硬件级

| 机制 | 说明 |
|------|------|
| 专用操作码 | 不同种类的 load/store 指令（如 acquire-load） |
| 地址阴影 | 利用虚拟地址高位编码操作类型 |
| 默认同步指令 | compare-and-swap、load-linked/store-conditional 默认视为同步 |
| 翻译为栅障 | 编译器将高层的类型信息翻译为底层的 fence 指令 |

### 7.4 正确标记（PL）方法

与 RCsc 同时提出。程序员只需确保：
- 所有参与竞争的普通操作都被标记为特殊（acquire/release）
- 进入临界区用 acquire，离开用 release

满足此条件后，程序行为等价于顺序一致性。

## 8. 讨论

### 8.1 关键权衡

| 方面 | SC | 宽松模型 |
|------|----|---------|
| 编程模型 | 简单直观 | 复杂（尤其以系统为中心时） |
| 硬件性能 | 受限 | 大幅提升（延迟隐藏） |
| 编译器优化 | 受限 | 接近单处理器的自由度 |
| 可移植性 | 高 | 不同架构差异大 |

### 8.2 DRF0 的重要意义

DRF0（以及更一般的"以程序员为中心"方法）将宽松模型的性能优势与 SC 的编程简易性统一了起来——程序员只需正确标记同步操作，无需理解底层重排细节。**现代 C++11 内存模型、Rust 的 Ordering、Java 的 volatile 和锁语义，本质上都是 DRF0 框架的后继者和具体实现。**

### 8.3 开放问题

- 是否存在一个"最佳"一致性模型，能同时满足可编程性、性能和可移植性？
- 语言设计者与硬件设计者的协作是否足够紧密？
- High Performance Fortran 的 forall（copy-in/copy-out 语义）展示了另一种高级模型的思路
