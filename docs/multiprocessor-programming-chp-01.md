## 1. 并发编程的挑战

### 1.1 背景与动机

早期计算机性能提升依赖摩尔定律——时钟频率逐年攀升，单线程程序自然变快。但约 2000 年代中期，功耗墙（power wall）导致这一趋势终结，处理器厂商转向多核架构。程序员不能再指望单线程性能的自动提升，必须学习编写并发程序才能充分利用硬件。

### 1.2 核心难点

1. **正确性（Correctness）** — 并发程序的不确定性（non-determinism）使得 bug 难以复现和调试。同一段并发代码可能运行一万次都正确，但在某一特定交错执行（interleaving）下出错。这被称为 **Heisenbugs**——观察时消失、不观察时出现的 bug。

2. **原子性缺失（Lack of Atomicity）** — 对共享变量的操作（如 `x++`）在机器层面由多条指令组成（读-改-写）。线程切换可能发生在任意两条指令之间，导致中间状态被其他线程看到，产生 **数据竞争（data race）**。

3. **可见性（Visibility）** — 现代处理器为了性能会重排指令（instruction reordering），并且每个核心有自己的缓存。一个线程对变量的修改，在其他线程中可能不可见，直到某个同步事件（如内存屏障）发生。这引出了 **内存一致性模型（memory consistency model）** 的概念。

4. **复合操作的原子性** — 即使单个操作是原子的，多个操作组成的复合操作（如 check-then-act、read-modify-write）也需要额外同步。催生了 **锁（lock）** 和 **事务内存（transactional memory）** 等抽象。

5. **性能和可扩展性（Performance & Scalability）** — 并发不总是带来加速。锁竞争（lock contention）、缓存乒乓（cache ping-pong）、false sharing 等问题可能导致性能退化。

### 1.3 三条核心原则

- **安全性（Safety）**：坏的事情永远不会发生（如数据竞争、死锁）
- **活性（Liveness）**：好的事情最终会发生（如无锁线程最终能进入临界区）
- **公平性（Fairness）**：每个线程都有机会推进（防止饥饿 starvation）

**核心论点**：并发编程本质上是难以推理的，因为执行交错的组合数量是指数级的，手动枚举所有可能交错不可行。需要形式化的推理工具和经过验证的并发抽象（锁、无锁数据结构、原子变量等），而不是靠直觉或"碰运气"来编写并发代码。

## 2. 共享内存模型

### 2.1 基本概念

共享内存模型是并发编程最直观的抽象。多个线程共享同一地址空间，通过读写共享变量来通信和协调。

```
线程 A ──→ 读/写 ──→ [共享内存] ←── 读/写 ←── 线程 B
```

### 2.2 核心要素

1. **线程（Thread）** — 独立的执行流，每个线程有自己的程序计数器、寄存器和调用栈，但共享堆（heap）和全局变量。

2. **共享对象（Shared Object）** — 被多个线程同时访问的数据结构。最简单的共享对象是共享整数，复杂的有队列、栈、哈希表等。

3. **寄存器（Register）** — 共享内存抽象为寄存器，定义了不同等级：
   - **安全寄存器（Safe Register）**：读操作返回的值要么是某个并发写操作正在写入的值，要么是最近完成的写操作的值。
   - **正则寄存器（Regular Register）**：安全 + 非重叠写不会返回旧值。
   - **原子寄存器（Atomic Register）**：所有操作满足线性一致性（linearizability）——每个操作在调用和返回之间的某个时刻瞬间完成。

### 2.3 内存一致性模型

处理器和编译器会重排指令以提高性能，这使得线程观察到的操作顺序可能与程序顺序不一致。一致性模型定义了线程对共享内存访问的排序保证：

- **顺序一致性（Sequential Consistency, SC）** — Lamport 1979 年提出。所有线程的执行结果等价于所有操作按某种全局顺序执行，且每个线程内部的程序顺序保持不变。现代处理器几乎都不原生支持 SC。

- **宽松一致性（Relaxed Consistency）** — 允许某些重排，通过 **内存屏障（memory fence / barrier）** 指令在必要时强制排序。不同的 ISA（x86 TSO、ARM、RISC-V）有不同的放松程度。

### 2.4 读-改-写（RMW）原子操作

仅有共享内存的读/写操作无法直接实现互斥，需要更强的 RMW 原子操作。

| 操作 | 语义 |
|------|------|
| **test-and-set (TAS)** | 读取旧值并写入新值，整体原子 |
| **compare-and-swap (CAS)** | 只有当前值等于期望值时才更新，否则返回实际值 |
| **fetch-and-add (FAA)** | 原子地增加一个值并返回旧值 |

### 2.5 关键洞察

共享内存模型虽然抽象简单，但硬件、编译器、语言层层叠加的复杂性使得"共享一个变量"变得极其微妙。核心挑战始终是：如何在允许尽可能多指令重排的前提下，保证关键操作的原子性和可见性。

## 3. 互斥

### 3.1 问题引入

两个线程各自执行 `counter = counter + 1`，机器层面需要三条指令：
1. `load counter, %reg`
2. `add 1, %reg`
3. `store %reg, counter`

若交错为 `A-load, A-add, B-load, B-add, B-store, A-store`，则 counter 最终只增加 1——**lost update（更新丢失）**。

### 3.2 临界区（Critical Section）

将对共享资源的访问代码段用互斥原语保护，称为临界区。任何时候最多只能有一个线程在临界区中。

### 3.3 正确互斥协议的三性质

| 性质 | 含义 |
|------|------|
| **互斥（Mutual Exclusion）** | 同一时刻至多一个线程在临界区中 |
| **无死锁（Deadlock-Free）** | 若干线程同时尝试进入临界区时，某个线程最终能成功 |
| **无饥饿（Starvation-Free）** | 每个尝试进入临界区的线程最终都能成功 |

无饥饿 ⇒ 无死锁，但反之不成立。

### 3.4 Lock / Unlock 抽象

```cpp
class Lock {
public:
    virtual void lock() = 0;
    virtual void unlock() = 0;
};

// 使用模式
lock.lock();
// 临界区操作共享资源
lock.unlock();
```

### 3.5 关键洞察

仅靠共享内存的读和写操作无法实现互斥。证明思路：如果互斥锁只使用读和写，那么当一个线程在临界区中时，另一个线程无法区分"第一个线程还在临界区"和"第一个线程崩溃/被调度出去"。

### 3.6 互斥的代价

- 串行化（serialization）：临界区只能串行执行
- 上下文切换和缓存失效
- 死锁、活锁、优先级反转等风险
- Amdahl 定律量化了串行部分对整体加速比的限制

## 4. 互斥算法

### 4.1 Peterson 算法（双线程互斥）

1981 年由 Gary Peterson 提出，仅使用共享内存的读/写操作，不依赖硬件原子指令。

```cpp
class Peterson implements Lock {
    private boolean[] flag = new boolean[2];
    private int victim;

    public void lock() {
        int i = ThreadID.get();
        int j = 1 - i;
        flag[i] = true;
        victim = i;
        while (flag[j] && victim == i) {}
    }

    public void unlock() {
        flag[ThreadID.get()] = false;
    }
}
```

**算法思想**：
- `flag[i]`：线程 i 表达进入意愿。
- `victim`：当前让行的线程——谁最后设置 victim，谁就礼让。

**正确性证明**：
- **互斥性**：假设两个线程同时进入临界区，则两者都通过了 `while` 条件检查。但总有一个线程最后设置 `victim`，导致它必须等待——矛盾。
- **无死锁**：如果只有一个线程试图进入，直接通过。如果两者都试图进入，至少有一个会设置 `victim` 为对方 ID，从而进入。
- **无饥饿**：线程 B 进出临界区后执行 `unlock()`（`flag[B] = false`），下次 B 再试图进入时，如果 A 还在等待，B 会设置 `victim = B`，使得 A 通过。

**历史意义**：证明了只用读写操作就能实现双线程互斥。

### 4.2 Filter 算法（N 线程互斥）

Peterson 算法到 N 线程的直接推广，构建 N-1 层的过滤阶梯。

```cpp
class Filter implements Lock {
    int[] level;     // level[thread] = 线程当前所在层号
    int[] victim;    // victim[layer] = 该层当前"让行者"

    Filter(int n) {
        level  = new int[n];    // 初始全 0（0 表示未参与竞争）
        victim = new int[n];    // victim[0] 不使用
    }

    void lock() {
        int me = ThreadID.get();
        for (int layer = 1; layer < n; layer++) {    // 共 n-1 层
            level[me] = layer;                       // [1] 我到达第 layer 层
            victim[layer] = me;                      // [2] 我声明让行

            for (int other = 0; other < n; other++) {
                if (other == me) continue;           // 跳过自己
                while (level[other] >= layer
                    && victim[layer] == me) {}       // [3] 等待条件
            }
        }
    }

    void unlock() {
        level[ThreadID.get()] = 0;   // [4] 归零，退出竞争
    }
}
```

**核心思想**：

- 共 N-1 层（从 1 到 N-1，第 0 层是未竞争状态）
- 层 k 至多能容纳 N - k + 1 个线程
- 通过第 N-1 层的线程唯一获得进入临界区的资格

**性质**：互斥、无死锁、无饥饿。

> *有 N 层楼，N 个人从第 1 层开始闯。每层最后进来的那个人被 victim 机制困在该层的等待区，其他人继续往上。最终 1 个人到达顶层进入临界区。当顶层的人离开（unlock）时，他就从塔里消失（level 归零），塔里的人数减 1。困在第 N-1 层等待区的人发现楼上没人了，就闯进顶层，然后依次往下连锁触发，所有被困住的人都能往上爬一层。重复上述过程，最终所有人都能到达顶层并离开。*

### 4.3 Bakery 算法（面包店算法）

Lamport 1974 年提出，真正实现了 N 线程互斥，对称且完全基于读写。

```cpp
class Bakery implements Lock {
    private boolean[] flag;
    private int[] label;

    public Bakery(int n) {
        flag = new boolean[n];
        label = new int[n];
    }

    public void lock() {
        int me = ThreadID.get();
        flag[me] = true;
        label[me] = max(label[0..n-1]) + 1;
        for (int k = 0; k < n; k++) {
            while (flag[k] && (label[k], k) < (label[me], me)) {}
        }
    }

    public void unlock() {
        flag[ThreadID.get()] = false;
    }
}
```

**算法思想——类比面包店**：
- 每个顾客进门时取号（`label[me] = max + 1`）
- 票号最小的顾客先服务，票号相同时编号小的优先
- 出店后清空自己的 flag

**关键设计点**：
- 取号操作不是原子的，两个线程可能拿到相同票号，用线程 ID 作为 tie-breaker
- `(label[k], k) < (label[me], me)` — 先比票号，再比 ID

**性质**：互斥、无死锁、无饥饿（FIFO 顺序）。

### 4.4 算法对比

| 算法 | 线程数 | 互斥 | 无死锁 | 无饥饿 | 空间复杂度 |
|------|--------|------|--------|--------|-----------|
| Peterson | 2 | ✓ | ✓ | ✓ | O(1) |
| Filter | N | ✓ | ✓ | ✓ | O(N) |
| Bakery | N | ✓ | ✓ | ✓(FIFO) | O(N) |

**书中强调**：这些算法证明了仅靠共享内存的读/写足以实现任意数量线程的互斥。但均使用忙等待（busy waiting），浪费 CPU。实际系统中使用操作系统提供的互斥量或硬件支持的 lock-free 原语。

## 5. 现代处理器架构的影响

### 5.1 指令重排

处理器和编译器对无数据依赖的指令进行重排。经典弱排序问题：

```
// 线程 1                        // 线程 2
data = 42;                       while (!flag) {}
flag = true;                     print(data);
```

在 SC 下若线程 2 看到 `flag == true`，则一定也能看到 `data == 42`。但由于写缓冲和乱序执行，线程 1 的 `flag = true` 可能先于 `data = 42` 被其他核心看到。

### 5.2 缓存一致性协议

现代多核处理器采用三级缓存架构。MESI 协议四个状态：

| 状态 | 含义 |
|------|------|
| **M（Modified）** | 已修改，与其他核心不一致 |
| **E（Exclusive）** | 只在本核心缓存中，与内存一致 |
| **S（Shared）** | 可能存在于多个核心缓存中，与内存一致 |
| **I（Invalid）** | 缓存行无效，需重新加载 |

缓存一致性只维护单缓存行的一致性，不维护多缓存行之间的相对顺序。

### 5.3 内存屏障（Memory Barrier / Fence）

| 类型 | 语义 |
|------|------|
| **StoreLoad** | 等待所有 store 写穿到内存后再执行后续 load |
| **LoadLoad** | 后续 load 之前先完成之前的 load |
| **StoreStore** | 后续 store 之前先完成之前的 store |
| **LoadStore** | 后续 store 之前先完成之前的 load |

x86/64（TSO 模型）上只有 **StoreLoad** 是真正的屏障。ARM/POWER 等弱一致性模型上四种屏障都需要。

### 5.4 对互斥算法的影响

Peterson、Bakery 等算法在 SC 下证明正确，但在现代处理器上会直接失效，因为指令重排可能破坏算法的排序假设。**仅仅实现算法逻辑是不够的，还需要在正确的位置插入内存屏障。**

### 5.5 内存一致性模型层级

| 层级 | 模型举例 |
|------|----------|
| x86 TSO | Total Store Order，较严格 |
| ARMv8 / POWER | Weak/Relaxed，允许大量重排 |
| C++11 / Rust | Happens-Before + Acquire-Release 语义 |
| Java | JMM（Java Memory Model） |

**核心教训**：并发算法必须在给定的内存模型下证明正确性。依靠直觉或不成立的假设写出的并发代码在生产环境中可能以极低概率出错，极难调试。

## 6. 公平性与活锁/死锁

### 6.1 安全性（Safety）与活性（Liveness）

| 类别 | 定义 | 例子 |
|------|------|------|
| **Safety** | 坏的事永远不会发生 | 互斥、无数据竞争、无死锁 |
| **Liveness** | 好的事最终会发生 | 无饥饿、无活锁、终止性 |

Safety 可通过有限执行序列检查，Liveness 需要看无限执行序列。

### 6.2 死锁（Deadlock）

多个线程互相等待对方释放资源，导致所有无法推进。

**死锁四个必要条件（Coffman 条件）**：
1. 互斥（Mutual Exclusion）
2. 持有并等待（Hold and Wait）
3. 不可抢占（No Preemption）
4. 循环等待（Circular Wait）

**无死锁（Deadlock-Free）**：至少有一个线程能成功进入临界区，但其他线程可能永远被阻塞（饥饿）。

### 6.3 饥饿（Starvation）与无饥饿（Starvation-Free）

**饥饿**：线程无限期无法获得所需资源（尽管其他线程在推进）。
**无饥饿（Starvation-Free）**：每个尝试进入临界区的线程最终都能成功。

无饥饿 ⇒ 无死锁，反之不成立。

### 6.4 活锁（Livelock）

线程在执行但做的是无用功，整体没有进展。例如两个线程反复互相谦让，无人进入临界区。

| 对比 | 死锁 | 活锁 |
|------|------|------|
| 线程状态 | 被阻塞，不执行 | 在执行，但做无用功 |

### 6.5 公平性度量

| 级别 | 含义 |
|------|------|
| **无死锁** | 至少一个线程能进入临界区 |
| **无饥饿** | 每个最终能运行的线程都能进入临界区 |
| **k-有界等待** | 某线程进入后，另一线程最多再等 k 次 |
| **FCFS / FIFO** | 先请求的先进入 |

### 6.6 权衡

追求最强公平性（FCFS）会引入更多同步开销和更低吞吐量。实际操作系统中的锁实现（如 Linux futex、pthreads mutex）是硬件原语、队列机制和公平性折衷的混合方案。

## 7. Amdahl 定律

### 7.1 公式

Amdahl 定律由 Gene Amdahl 在 1967 年提出：

```
加速比 S = 1 / ((1 - P) + P/N)
```

- **P** = 程序中可并行化的比例
- **N** = 处理器核心数量
- **1 - P** = 必须串行执行的比例

### 7.2 核心结论

当 N → ∞ 时，S → 1 / (1 - P)。**即使有无限多的核心，加速比的上限也被串行部分所限制。**

### 7.3 具体示例

| 串行比例 | 2 核心 | 4 核心 | 16 核心 | ∞ 核心 |
|:--------:|:------:|:------:|:-------:|:------:|
| 0% | 2.00x | 4.00x | 16.00x | ∞ |
| 5% | 1.90x | 3.48x | 9.14x | 20.00x |
| 10% | 1.82x | 3.08x | 6.40x | 10.00x |
| 25% | 1.60x | 2.29x | 3.37x | 4.00x |
| 50% | 1.33x | 1.60x | 1.88x | 2.00x |

### 7.4 对并发编程的影响

1. **锁就是串行化** — 每次上锁/解锁都是串行执行，是 Amdahl 定律中的 `(1-P)` 部分。
2. **可扩展性取决于临界区的比例** — 95% 并行时，20 核心的加速比不到 10 倍。
3. **细粒度锁好于粗粒度锁** — 减少临界区粒度和持有时间以提升可扩展性。
4. **无锁数据结构不免除 Amdahl 定律** — CAS 失败重试也是串行化开销。

### 7.5 Gustafson 定律（补充视角）

```
加速比 S = (1 - P) + N × P
```

Gustafson 的观察：实际问题规模通常随核心数增长，等规模下的加速比比 Amdahl 预测的更乐观。两者视角互补。

## 8. 总结

第一章作为全书的引言，覆盖了以下核心内容：

1. **并发编程的挑战** — 超大规模交错、Heisenbug、原子性缺失、可见性问题，以及安全性/活性/公平性三条核心原则。
2. **共享内存模型** — 寄存器层次（Safe/Regular/Atomic）、内存一致性模型（SC vs Relaxed）、RMW 原子操作的必要性。
3. **互斥** — 临界区定义、三性质（互斥/无死锁/无饥饿）、仅靠读/写无法实现互斥。
4. **经典互斥算法** — Peterson（双线程）→ Filter（N 线程）→ Bakery（N 线程 FIFO），逐步递进。
5. **硬件约束** — 指令重排、MESI 缓存一致性、内存屏障、弱一致性模型及其对算法正确性的影响。
6. **公平性理论** — 死锁、饥饿、活锁的严格定义与公平性分类。
7. **Amdahl 定律** — 串行部分对可扩展性的限制，以及 Gustafson 定律的补充视角。
