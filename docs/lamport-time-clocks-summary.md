<h1 align="center">Time, Clocks, and the Ordering of Events in a Distributed System</h1>
<p align="center">Leslie Lamport</p>

## 1. 分布式系统的难题——不存在全局时钟

分布式系统里的多个进程跑在不同的机器上，彼此只靠发消息来通信。要协调这些进程，需要确定事件的先后顺序。最自然的想法是比较时间戳。

但问题在于，这些机器没有共享的全局时钟——每台机器有自己的本地时钟，但彼此不同步。A 的钟显示 10:00 时，B 的钟可能已经是 10:02，拿 A 的时间戳去和 B 的比先后没有意义。

那能不能先把时钟同步起来？也不行。同步需要发消息，但消息在网络上传输的时间是不确定的——可能是 5ms 也可能是 50ms，你永远测不准。要精确测量传输时间，本身就需要同步的时钟，这就形成了一个死循环，所以"比物理时间"这条路走不通。

Lamport 换了个思路：**不比时间，比因果关系。** 他把这种关系称为 **"happened before"（→）**——如果事件 A 有可能影响事件 B，那 A 就在 B 之前；如果不能，那就不分先后。

## 2. 事件顺序的定义——happened before 偏序

happened before（→） 定义如下：

* **同一进程内**：a 在程序顺序上先于 b，则 a → b；
* **消息发送与接收**：a 发送消息，b 接收 a 发送的消息，则 a → b；
* **传递性**：a → b 且 b → c，则 a → c。

通过 happened before 关系我们就能确定事件发生的先后顺序。

但如果两个事件不存在 happened before 关系，那么这两个事件就是**并发的**。并发的事件不分先后，也没必要分，关键在于 happened before 只排"有关系的"，不排"没关系的"，这就是它叫"偏序"的原因。

## 3. 偏序的实现——逻辑时钟

偏序是一个概念，但程序跑起来需要具体的数据结构。Lamport 的实现方案很简单：每个进程维护一个计数器，称为逻辑时钟。

但什么样的时钟才算"正确"？论文定义了一个标准——**时钟条件 (Clock Condition)**：如果 a 在因果上先于 b，那 a 的时钟值必须小于 b 的时钟值。

怎么实现时钟条件？论文给出了两条规则：

- **IR1：** 每发生一个事件，就递增逻辑时钟，事件的时间戳就是当前的时钟值。
- **IR2：** 发消息时，把当前时钟值写在消息上；收消息时，把自己的时钟更新为 `max(自己的值, 消息上的值 + 1)`。

IR1 和 IR2 保证了时钟条件成立，具体而言：

- 同一进程里后发生的事件时钟值一定更大（IR1）；
- 发消息事件的时钟值一定小于收消息事件的时钟值（IR2）；
- 若 a 的时钟值小于 b，b 的时钟值小于 c，那么 a 的时钟值一定小于 c（IR1+IR2）。

逻辑时钟不测量时间，它只编码因果关系——这也是它叫"逻辑"时钟而非"物理"时钟的原因。

## 4. 偏序的补全——全序化

逻辑时钟给出了每个事件的编号，但偏序有个麻烦：它不对并发事件排序——而很多问题需要全序，比如两台电脑抢打印机，总得有一个先用。

Lamport 的做法很直接：**先比编号，编号小的在前；编号相同，进程 ID 小的在前。**

这个规则就是全序。它和偏序的关系是：有因果关系的，编号一定不同（时钟值递增），自然不会排反；没有因果关系的，编号可能相同，那就用进程 ID 强行分先后。

全序不是唯一的——换一套时钟或换一种打破平局的方式，全序就可能不同。但偏序是唯一的，它是由事件的因果顺序决定的。

## 5. 全序的应用——分布式互斥算法

有了全序，就能解决一个经典问题：**多个进程怎么公平地抢一个共享资源（比如打印机）**，而且不靠中心调度器。

**三个目标：** 

* 已经拿到资源的必须释放后才能给别人；
* 请求按发出顺序被授予；
* 每个请求最终都能拿到。

**三个假设：** 

* 两台进程之间的消息按发送顺序到达（FIFO）；
* 每条消息最终会被收到；
* 每个进程可以直接给其他任意进程发消息。

每个进程都维护一个**本地队列**，初始只有一条消息——`T0:P0 requests resource`，意思是 P0 一开始就持有资源。队列里的条目按全序排序，所有进程的队列内容一致，这是整个算法的根基。

算法的核心分五步：

1. **想用资源时**，向所有人广播"我要用，时间戳 Tm"，同时把这条消息放进自己的队列；
2. **收到别人的请求时**，放进自己的队列，回一条确认消息；
3. **用完资源时**，从队列删掉自己的请求，向所有人广播"我用完了"；
4. **收到别人用完了**，从队列删掉对方的请求；
5. **什么时候真正拿资源？** 两个条件同时满足才行：① 自己的请求在队列里排在最前面；② 收到了所有其他进程的时间戳大于 Tm 的消息。

这个算法的思想可以推广到任何分布式同步问题：把所有操作按全序排序，每个进程独立执行状态机（定义相同、命令序列相同）。这就是后来 Paxos、Raft 等共识算法的基础——**复制状态机 (Replicated State Machine)**。

## 6. 逻辑时钟的缺陷——反常行为

全序在系统内部工作完美，但有一个漏洞：**系统外的因果关系**，全序感知不到。

举一个例子。你在北京的电脑 A 上发出请求 A，A 的逻辑时钟值是 5。然后你打电话告诉上海的朋友，让他在电脑 B 上发出请求 B。B 的时钟跑得慢，B 的逻辑时钟值只有 3。你的视角里，A 先 B 后；但系统的全序里，B（3）排在了 A（5）前面——这不合理。

问题出在哪？因为"打电话"这件事不在系统里。系统只知道收到了两个请求，编号分别是 3 和 5，但它不知道你打过电话，不知道 B 的请求是因为 A 才发起的。

**两种解法：** 

* 让用户负责：你发完 A 后把时间戳告诉朋友，朋友发 B 时要求一个更大的时间戳；
* 用物理时钟：如果 B 的物理时钟和 A 差不多准，那 B 发请求时的时间戳自然比 A 大，就不需要用户插手了。

## 7. 缺陷的修复——物理时钟同步

反常行为的根源是逻辑时钟和真实时间脱钩。论文引入**强时钟条件 (Strong Clock Condition)** 来解决这个问题：不仅系统内的事件，连打电话、当面说话这类外部事件，只要存在因果关系，时间戳也必须递增。

逻辑时钟做不到这点，但物理时钟可以。使用物理时钟需要满足两个基本条件：

* **PC1：** 时钟走速接近 1，误差不超过 κ（晶振时钟约 κ ≤ 10⁻⁶）；
* **PC2：** 任意两个时钟的读数差不超过 ε。

物理时钟的实现规则和逻辑时钟类似，关键区别在于收消息时不是简单 +1，而是加上已知的最短传输延迟 μm——真实世界消息不可能瞬间到达。每条消息有自己的 μm（由物理距离决定），全局的光速限制最小值称为 μ。

基于这些条件，论文推导出反常行为消除条件：当 `ε / (1 - κ) ≤ μ` 时，反常行为不可能发生。也就是说，时钟偏差足够小，外部因果顺序就能反映在时间戳上。

那么如何保证时钟偏差足够小？论文的核心定理给出了答案。设通信网络直径（最远两个进程的跳数）为 d，每 τ 秒每条线路至少发一条消息，消息的不可预测延迟不超过 ξ。经过足够长的时间后，时钟偏差的上界为：`ε = d(2κτ + ξ)`

通俗地说：时钟越不准（κ 越大）、通信越稀疏（τ 越大）、延迟抖动越大（ξ 越大）、网络跳数越多（d 越大），偏差就越大。但这个偏差是可控的、有界的。论文附录给出了完整的数学证明。

## 8. 总结

在没有全局时钟的分布式系统中，通过逻辑时钟和因果关系，可以在偏序之上构建一个与因果一致的全序，从而解决同步问题。
